Cloud Concepts 
Designing a Secure and Scalable Web Application on AWS
A startup looking to host a web application for customers across two countries faces a common but critical challenge: how do you build something that is reliable, secure, and capable of growing without rebuilding it from scratch every time? AWS provides exactly the tools needed to solve this problem. This essay proposes an architecture that addresses all three requirements.
Cloud Computing Benefits
Before diving into the architecture, it is worth understanding why cloud computing is the right choice for a startup in this position. Traditional on-premises infrastructure requires significant upfront investment in hardware, maintenance staff, and physical space. Cloud computing eliminates these barriers entirely. With AWS, the startup pays only for what it uses, can scale up or down based on demand, and benefits from AWS's globally distributed infrastructure without managing a single physical server. For a startup serving two countries, this flexibility is invaluable.
Regions and Availability Zones
AWS organises its infrastructure into Regions; physical locations around the world and Availability Zones (AZs), which are isolated data centres within each region. For a startup serving two countries, deploying in a region closest to the majority of users minimises latency. For example, using the EU (London) or US East (N. Virginia) region depending on where the customers are located. Within that region, deploying across at least two Availability Zones ensures that if one data centre experiences an outage, the application continues running in the other. This is the foundation of high availability.
EC2 for Compute
Amazon EC2 (Elastic Compute Cloud) provides the virtual servers that run the application. For this startup, EC2 instances would be deployed in both Availability Zones behind a Load Balancer, which distributes incoming traffic evenly across both instances. This means no single server becomes a bottleneck, and if one instance fails, traffic is automatically routed to the healthy one. The startup can start with small instance types like t3.micro and scale up as the user base grows without any downtime.
S3 for Storage
Amazon S3 (Simple Storage Service) serves two purposes in this architecture. First, it stores static assets such as images, CSS files, and JavaScript offloading this from the EC2 instances and reducing their workload. Second, it stores application backups. S3 is designed for 99.999999999% durability, meaning data is virtually never lost. Enabling versioning on the backup bucket ensures that even if a file is accidentally overwritten, previous versions can be restored.
IAM Users vs Roles
Identity and Access Management (IAM) controls who can access what in AWS. IAM users are individual accounts with permanent credentials, suitable for human team members who need console access. IAM roles, on the other hand, are temporary credentials assigned to services; for example, giving an EC2 instance permission to read from S3 without storing any passwords on the server itself. For this startup, each team member should have their own IAM user with only the permissions they need, following the principle of least privilege. EC2 instances should use IAM roles rather than hardcoded credentials.
Security Groups
Security Groups act as virtual firewalls around EC2 instances. For the web server, the security group should allow inbound HTTP traffic on port 80 and HTTPS on port 443 from anywhere, since it is a public facing application. SSH access on port 22 should be restricted to specific IP addresses, never open to the entire internet. All outbound traffic can be allowed so the server can communicate with external services as needed.
SSH Key Access Best Practices
SSH key pairs are used to securely connect to EC2 instances. The private key should never be shared, stored in version control, or sent over email. Each team member should have their own key pair. Additionally, SSH access should be restricted by IP address in the Security Group, and for extra security, a Bastion Host, a single hardened server that acts as the only entry point for SSH should be used rather than exposing all instances directly.
Conclusion
This architecture — EC2 instances across multiple Availability Zones, static assets and backups in S3, strict IAM policies, and properly configured Security Groups gives the startup a reliable, secure, and scalable foundation. As the business grows, additional services like CloudFront for content delivery and RDS for managed databases can be added without redesigning the core architecture.

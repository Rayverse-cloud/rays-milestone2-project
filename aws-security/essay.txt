AWS Security Review 
Identifying and Fixing Poor AWS Security Practices
The scenario described — a company using the root account for daily operations, allowing SSH from anywhere, storing backups without access controls, and sharing one login across the team represents a collection of serious security risks that are unfortunately common in early stage organisations. This essay identifies each risk and recommends concrete improvements.
Risk 1 — Using the Root Account for Daily Operations
The AWS root account has unrestricted access to every service and setting in the account, including the ability to delete everything, change billing details, and close the account entirely. Using it for daily operations means that if the credentials are ever compromised, the attacker has complete control. The root account should be used only for the initial account setup and then locked away. Multi-Factor Authentication (MFA) should be enabled on it immediately. All daily work should be done through individual IAM users with only the permissions they actually need.
Risk 2 — SSH Open to 0.0.0.0/0
Allowing SSH from anywhere means every device on the internet can attempt to connect to the company's servers on port 22. Automated bots constantly scan the internet for open SSH ports and run brute force attacks trying thousands of password combinations per minute. The fix is simple but critical: restrict the SSH inbound rule in the Security Group to specific trusted IP addresses only, using /32 notation for individual IPs. Additionally, password authentication should be disabled entirely only SSH key pairs should be accepted.
Risk 3 — S3 Backups Without Access Controls
Storing backups in S3 without proper access controls risks either exposing sensitive data publicly or allowing any team member to accidentally delete or overwrite critical backups. S3 buckets should never be public unless they are specifically serving public static assets. Backup buckets should be private, with access granted only through IAM roles to the specific services that need it. Bucket versioning should be enabled so deleted or overwritten files can be recovered. S3 Object Lock can be used to make backups immutable for a defined period.
Risk 4 — Shared Login Across the Team
Sharing one set of credentials across the team makes it impossible to track who did what, and means that when one person leaves the company, changing the password disrupts everyone. Every team member must have their own IAM user account. Permissions should follow the principle of least privilege, a developer should not have access to billing, and a finance person should not have access to EC2. AWS CloudTrail should be enabled to log every action taken in the account, creating a full audit trail.
How IAM, Roles, and Security Groups Should Be Used
IAM users should be created for every human who needs access, with permissions granted through IAM policies scoped to only what is necessary. IAM roles should be used for services; for example, an EC2 instance that needs to read from S3 should be assigned a role with that specific permission rather than having access keys hardcoded into the application. Security Groups should follow a least-access principle, only the ports and IP ranges that are genuinely needed should be open, and rules should be reviewed regularly.
Conclusion
The risks in this scenario are significant but entirely fixable. Locking down the root account, creating individual IAM users, restricting SSH to known IPs, securing S3 buckets with proper policies and versioning, and eliminating shared credentials together form a much stronger security posture. Security is not a one-time task, it requires regular review as the organisation grows and its AWS usage evolves.
